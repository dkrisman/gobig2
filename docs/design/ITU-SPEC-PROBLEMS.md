# ITU sample encoder deviations from the JBIG2 spec

The ITU/ISO **conformance corpus** for JBIG2 (T.88 / ISO/IEC
14492 Annex A,
[`JBIG2_ConformanceData-A20180829`](https://www.itu.int/rec/T-REC-T.88))
ships with **TT1 through TT10**, each generated by the ICT-Link
Corporation's reference encoder
(`JBIG2_SampleSoftware-A20180829`, BSD-3, written by Shigetaka
Ogawa). When that corpus is run through any open-source JBIG2
decoder you can find - gobig2, pdfbox-jbig2, PDFium - only
**TT1, TT9, TT10** decode cleanly. **TT2-TT8 fail** in every
implementation.

This document explains why. The short version: the corpus
encoder writes spec-deviating shapes that no real-world JBIG2
producer (Acrobat, jbig2enc, scanner firmware, mobile OCR
toolchains) emits. The decoders that exist in the wild were
hardened against the streams the wild actually produces, not
against the corpus.

It's also the story of what we tried, why we stopped, and the
file/line numbers a future contributor would need to take
another swing at it.

## TL;DR

| Decoder | TT1 | TT2 | TT3 | TT4 | TT5 | TT6 | TT7 | TT8 | TT9 | TT10 |
|---|---|---|---|---|---|---|---|---|---|---|
| ICT-Link sample (the corpus author's own) | Y | Y | Y | Y | Y | Y | Y | Y | Y | Y |
| pdfbox-jbig2 (Apache 2.0) | ~ | N | N | N | N | N | N | N | Y | Y |
| PDFium (BSD-3, Chromium) | ~ | blank | blank | blank | blank | blank | blank | blank | Y | Y |
| **gobig2** | ~ | N | N | N | N | N | N | N | Y | Y |

`~` = decoded but with the documented 74-pixel encoder-lossy
delta against the reference (see TT1 entry below).

## Why "the corpus encoder is non-spec" instead of "everyone else
got it wrong"

The conformance suite is supposed to *measure* spec compliance.
Instead, the encoder it ships diverges from T.88 in several
specific places, and the decoder that ships with the same
package mirrors those divergences. The fixtures encode -> decode
internally with **0 distortion** (verified: built and ran
`make test` against the unmodified ICT-Link source - every TT
passes Distortion=0.000000). They DON'T encode -> decode against
any other reader.

That's the inverse of what a conformance suite should test. The
suite tests **the encoder's own decoder**, not "any spec-compliant
decoder." Chromium / Acrobat / jbig2enc / pdfbox / gobig2 all
follow the spec; the corpus encoder follows itself. Where they
disagree, the conformance suite calls everyone else wrong.

## The deviations, one by one

All references below are to source files in
`JBIG2_SampleSoftware-A20180829/source/`.

### 1. TT2: Huffman symbol-dict with `BmSize > 0` writes per-symbol T.6 streams

**Spec (T.88 §6.5.5.4):** in SDHUFF=1 mode, when SDHUFFBMSIZE is
set, decode the BMSIZE field - that's the **byte count** of a
single CCITT Group 4 (T.6 / MMR) stream covering the
**collective bitmap** of `TotalWidth x Height` for the height
class. Split the decoded collective bitmap into individual
symbol bitmaps via the per-symbol widths.

**ICT-Link encoder ([Jbig2ENC.cpp:470-476](https://www.itu.int/rec/T-REC-T.88)):**

```cpp
if (SymbolDic->BmSize) {
    str = JBIG2_HuffEnc(SymbolDic->BmSize, str, &Huff[B_No_Huff]);
    str = ByteStuffOutJXR(str);
    for (jjj = 0; jjj < numWidth[kkk]; jjj++, ...) {
        ImageSym = ...;
        Image = ImageSym->Image;
        str = T4T6Encmain(str, Image, 0, T6, 0); // T.6 PER symbol
    }
}
```

The encoder writes the BMSIZE field with whatever value the user
configured (`-Huff_BMSize 1` => BmSize=1 in TT2's case), then
emits **N independent T.6 streams** - one per symbol of this
height class - each byte-aligned via `ByteStuffOutJXR`. The
BMSIZE field has lost its spec meaning; it's now a non-zero
**flag** ("use per-symbol T.6") rather than a byte count.

**ICT-Link decoder ([Jbig2DEC.cpp:845-857]):** mirrors with one
`T4T6DecMain(..., Width[k], Height)` call per symbol, ignoring
the BMSIZE field's numeric value entirely.

**Why this is wrong by the spec.** Two T.6 streams cannot be
read as one - the per-row state inside T.6 is fresh per stream,
and T.6's two-dimensional codes reference the *previous* row in
the same stream. Concatenating two T.6 streams of width A and B
and decoding as one stream of width A+B produces garbage. So a
spec-literal decoder reading BMSIZE bytes as one collective MMR
gets the wrong bitmap (and probably runs off the end of the
declared bytes).

### 2. TT2: No EXFLAGS Huffman section - just two zero bytes

**Spec (T.88 §6.5.10):**

> 2) Decode a value using Table B.1 if SDHUFF is 1, or the IAEX
>    integer arithmetic decoding procedure if SDHUFF is 0. Let
>    EXRUNLENGTH be the decoded value.
> 3) Set EXFLAGS[EXINDEX..EXINDEX+EXRUNLENGTH-1] to CUREXFLAG.
> 4) EXINDEX += EXRUNLENGTH; toggle CUREXFLAG.
> 5) Repeat 2)-4) until EXINDEX = SDNUMINSYMS + SDNUMNEWSYMS.

The spec is unambiguous: even when SDHUFF=1, decode EXFLAGS via
Huffman B.1.

**ICT-Link encoder ([Jbig2ENC.cpp:494-495]):**

```cpp
str = Stream1ByteWrite(str, 0, str->buf_length);
str = Stream1ByteWrite(str, 0, str->buf_length);
```

After the height-class loop, the encoder writes **two literal
zero bytes**. No Huffman B.1 encoding. The value of those bytes
isn't even an encoding of "all symbols exported" - it's just
filler.

**ICT-Link decoder ([Jbig2DEC.cpp:891-892]):**

```cpp
Ref_1Byte(str);
Ref_1Byte(str);
```

Two raw byte reads, no Huffman parsing. The decoder *implicitly*
treats every new symbol as exported; the export-flag array is
never built.

**Why this matters.** A spec-literal decoder calls Huffman B.1's
`DecodeAValue` against bytes that aren't Huffman-encoded.
Sixteen zero bits decode as a long string of `EXRUNLENGTH = 0`
values, never advancing EXINDEX. With our defensive "stuck at
zero => no progress" guard, gobig2 errors out cleanly. Without
that guard (the original behavior across the JBIG2 ecosystem),
decoders either spin forever or eventually return some
EXRUNLENGTH > total and trip an out-of-bounds.

### 3. TT3: Arithmetic symbol-dict + the 2-IAEX shortcut

**Spec:** same §6.5.10 - loop on IAEX until EXINDEX reaches
total.

**ICT-Link decoder ([Jbig2DEC.cpp:925-926]):**

```cpp
tempD1 = MQ_DecInteger(str, codec, IAEX, MQ_Eaddr, &Flag); // IAEX
tempD2 = MQ_DecInteger(str, codec, IAEX, MQ_Eaddr, &Flag); // IAEX
```

Hardcoded to two IAEX reads, regardless of `SDNUMINSYMS +
SDNUMNEWSYMS`. Equivalent to "every symbol is exported" in the
limit case. Spec-literal decoders that loop on EXINDEX
sometimes happen to land at exactly two reads (for the all-
exported case), so they appear to work - until they don't.

### 4. TT4: Generic-region template 1

**Param file (`jbig2_Param4.ini`):** `-Template 1` instead of
the default 0.

This isn't a deviation per se; template 1 is in the spec
(§6.2.5.4). But it's a less-tested code path. Our decoder
handles template 1 in `internal/generic/templates.go`, so the
arithmetic path through TT4's symbol bitmaps works in
isolation. But TT4 also has the BMSIZE/EXFLAGS issues from
TT2-TT3 (it shares those param settings), so it fails for
the same reasons.

### 5. TT5: SymbolDictionary refinement-defined symbols

**Param file (`jbig2_Param5.ini`):** uses `-Ref` syntax to
define a symbol as a refinement of a previous one in the same
dictionary. Spec §6.5.8.2 covers this; the encoder writes a
refinement-region embedded inside the symbol dictionary.

We support this for LibGfx fixtures (which ship multiple
`bitmap-symbol-symhuffrefine*.jbig2` cases that pass our
107/107 LibGfx run). TT5 uses an additional encoder-specific
shape that the LibGfx fixtures don't, plus the EXFLAGS issue.

### 6. TT6, TT7, TT8: Text region + refinement + color extension

**Param files:** TT6 and TT8 use `-RefCorner 1`,
`-RefTemplate 1`, refinement instances on the text region.
TT8 additionally uses `-ColorExt`, the spec's optional color
extension annex.

All three exercise text-region paths past the symbol-dict.
Since the symbol dict already fails for the EXFLAGS reasons
above, the text region never gets a chance. If the symbol dict
were fixable, these would still need their own encoder-deviation
analysis.

## Why a one-line fix breaks LibGfx

The temptation is to just match the ICT-Link conventions - they
ARE the corpus author's, after all. We tried it. The result:

> A first-pass implementation that swapped conventions wholesale
> (per-symbol T.6 + skip EXFLAGS, mark all exported) broke 7
> LibGfx fixtures.

The fixtures it broke:

- `bitmap-symbol-symhuff-texthuff.jbig2`
- `bitmap-symbol-symhuff-texthuffB10B13.jbig2`
- `bitmap-symbol-symhuffB5B3-texthuffB7B9B12.jbig2`
- `bitmap-symbol-symhuffcustom-texthuffcustom.jbig2`
- `bitmap-symbol-symhuffrefine-textrefine-export.jbig2`
- `bitmap-symbol-symhuffrefine-textrefine.jbig2`
- `bitmap-symbol-symhuffrefineseveral.jbig2`

These all use SDHUFF=1 + SDREFAGG=0 - the exact same shape as
TT2 - but the encoder that produced them (LibGfx's, or
jbig2enc's) followed the spec. EXFLAGS Huffman is real;
collective MMR is real. Switching to the ICT-Link convention
unconditionally tells those fixtures the wrong thing.

## What it would take to support both

Conceptually: detect at runtime which encoder produced the
bytes and switch the decoding strategy. The signals we have
to work with:

1. The segment's declared **DataLength** vs the bytes consumed
   so far. If after the height-class loop only 2 bytes remain
   in the segment, that's strong evidence of ICT-Link's "two
   trailing zero bytes" shape. But LibGfx EXFLAGS Huffman
   sometimes leaves 2 bytes of alignment padding too, so this
   alone isn't sufficient.
2. The actual **bytes** at the EXFLAGS section. If they are
   exactly `00 00`, that's even stronger evidence - but a
   spec-literal "all symbols exported" with N=8 encodes as
   B.1(0) || B.1(8) = `0` + `0000` || `0` + `1000` =
   `00000 01000` = padded `00100 00000` = `0x20 0x00`. A
   shape that's *almost* `00 00`. Heuristics get fragile.
3. The **BMSIZE** value. ICT-Link writes BmSize=1 (or whatever
   the user configured) as a marker; spec-literal collective
   MMR writes the actual byte count. A BMSIZE=1 with > 1 byte
   of remaining data in the segment is strong evidence of
   ICT-Link convention.

A robust implementation tracks all three and switches on a
weighted signal. We sketched it; it's not implemented.

## Why we (and Adobe, Chromium, Apache) all just don't

PDF readers in the wild see streams from:

- **Acrobat** (the original JBIG2 source). Spec-literal.
- **jbig2enc** (Apache 2.0, Adam Langley). Spec-literal.
- **Scanner firmware** (Canon, Xerox, HP). All Acrobat-derived.
- **Mobile OCR toolchains** (Tesseract, Google Cloud Vision).
  Spec-literal via libjbig2dec or jbig2enc.

None of them produces ICT-Link-shaped streams. The **only**
JBIG2 producer that does is the conformance corpus's own
encoder, and the only thing that consumes those streams is
its bundled decoder.

So the question for any decoder maintainer is: do we add
implementation complexity (runtime mode detection, two parser
paths through the symbol dict, careful test coverage to make
sure we don't break either) for a corpus that nobody in the
wild produces? The answer across the ecosystem has been **no**.
gobig2 sides with that consensus.

If you're a PDF reader integrator and you DO encounter a
JBIG2 stream that gobig2 fails on:

1. Open the stream with `gobig2 --inspect`. If the symbol
   dictionary's referred-to-segments and SDFLAGS look like
   TT2's pattern (SDHUFF=1, BMSIZE > 0, no SDREFAGG, EXFLAGS
   bytes look like `00 00`), it's the ICT-Link convention.
2. File a bug. We'll add the runtime detection.

But absent a real-world fixture, the implementation cost is
not worth carrying for the corpus alone.

spec-compliant decoders can achieve against this fixture
set.

PDF-extracted sample is the real-world correctness floor;
we're at the same place every other open-source JBIG2 decoder
sits.
